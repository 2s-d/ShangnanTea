# 订单状态体系设计文档

## 一、现状分析

### 1.1 当前状态定义

#### 订单状态（Order.status）
- `0`: 待付款 (STATUS_PENDING_PAYMENT)
- `1`: 待发货 (STATUS_PENDING_SHIPMENT)
- `2`: 待收货 (STATUS_PENDING_RECEIPT)
- `3`: 已完成 (STATUS_COMPLETED)
- `4`: 已取消 (STATUS_CANCELLED)
- `5`: 已退款 (STATUS_REFUNDED)

#### 支付单状态（Payment.status）
- `0`: 待支付 (STATUS_PENDING)
- `1`: 支付成功 (STATUS_SUCCESS)
- `2`: 支付失败 (STATUS_FAILED)
- `3`: 已关闭 (STATUS_CLOSED)

#### 退款状态（Order.refundStatus）
- `0`: 无退款申请
- `1`: 申请中
- `2`: 已同意
- `3`: 已拒绝

### 1.2 当前问题

1. **未付款订单显示"未知方式"**：订单创建时 `paymentMethod` 为空，但前端显示"未知方式"不合理
2. **状态冗余**：订单状态和支付单状态有重叠（待付款 vs 待支付）
3. **退款状态不明确**：退款申请中时，订单状态仍然是"待发货/待收货/已完成"，用户无法直观看到"退款中"
4. **超时未支付处理缺失**：订单创建后如果长时间未支付，没有自动取消机制
5. **支付失败后的处理不明确**：支付失败后订单状态如何处理

## 二、状态体系设计方案

### 2.1 核心设计原则

1. **订单状态为主，支付状态为辅**：订单状态是用户最关心的，支付状态作为辅助信息
2. **状态互斥**：同一时间订单只能处于一个主状态
3. **状态可追溯**：所有状态变更都有时间记录
4. **状态闭环**：每个状态都有明确的进入和退出路径

### 2.2 订单状态重新设计（7种状态）

| 状态值 | 状态名称 | 说明 | 可执行操作 |
|--------|---------|------|-----------|
| `0` | **待付款** | 订单已创建，等待用户支付 | 用户：支付、取消订单<br>系统：超时自动取消 |
| `1` | **待发货** | 已支付，等待商家发货 | 商家：发货<br>用户：申请退款 |
| `2` | **待收货** | 已发货，等待用户确认收货 | 用户：确认收货、申请退款<br>系统：超时自动确认 |
| `3` | **已完成** | 用户已确认收货 | 用户：评价、申请退款（限时） |
| `4` | **已取消** | 订单被取消（用户取消或超时取消） | 无操作 |
| `5` | **退款中** | 退款申请已提交，等待商家/管理员处理 | 商家/管理员：同意/拒绝退款<br>用户：查看退款进度 |
| `6` | **已退款** | 退款已处理完成 | 无操作 |

**关键变化**：
- 新增 `5: 退款中` 状态，替代原来用 `refundStatus=1` 但订单状态不变的情况
- `已退款` 从 `5` 改为 `6`（向后兼容需要迁移）

### 2.3 支付单状态保持不变（4种）

支付单状态独立管理，用于支付流程控制：

| 状态值 | 状态名称 | 说明 |
|--------|---------|------|
| `0` | 待支付 | 支付单已创建，等待支付 |
| `1` | 支付成功 | 支付已完成 |
| `2` | 支付失败 | 支付失败（余额不足、密码错误等） |
| `3` | 已关闭 | 支付单已关闭（超时、取消等） |

### 2.4 退款状态简化（仅用于退款流程细节）

退款状态保留，但仅作为退款申请的辅助信息：

| 状态值 | 状态名称 | 说明 |
|--------|---------|------|
| `0` | 无退款申请 | 默认状态 |
| `1` | 申请中 | 用户已提交退款申请 |
| `2` | 已同意 | 商家/管理员已同意退款 |
| `3` | 已拒绝 | 商家/管理员已拒绝退款 |

**注意**：当订单状态为 `5: 退款中` 时，`refundStatus` 必须为 `1`。

## 三、状态流转图

```
创建订单
   ↓
[0] 待付款 ←──────────────────┐
   ↓ (支付)                    │
   ├─→ 支付成功 → [1] 待发货   │
   ├─→ 支付失败 → [0] 待付款   │ (可重试支付)
   └─→ 超时/取消 → [4] 已取消  │
                              │
[1] 待发货                    │
   ↓ (发货)                   │
   ├─→ [2] 待收货             │
   └─→ 申请退款 → [5] 退款中   │
                              │
[2] 待收货                    │
   ↓ (确认收货)               │
   ├─→ [3] 已完成             │
   ├─→ 申请退款 → [5] 退款中   │
   └─→ 超时自动确认 → [3] 已完成│
                              │
[3] 已完成                    │
   ↓ (限时内可申请退款)        │
   └─→ 申请退款 → [5] 退款中   │
                              │
[5] 退款中                    │
   ↓ (处理退款)               │
   ├─→ 同意 → [6] 已退款 ──────┘
   └─→ 拒绝 → [1/2/3] 恢复原状态
              (根据退款前状态恢复)
```

## 四、关键业务规则

### 4.1 支付相关

1. **订单创建时**：
   - 订单状态 = `0: 待付款`
   - 创建支付单，状态 = `0: 待支付`
   - `paymentMethod` = 用户选择的支付方式（alipay/wechat），**即使未支付也要记录**
   - `paymentTime` = null

2. **支付成功时**：
   - 支付单状态 = `1: 支付成功`
   - 订单状态 = `0` → `1: 待发货`
   - `paymentTime` = 当前时间

3. **支付失败时**：
   - 支付单状态 = `2: 支付失败`
   - 订单状态保持 `0: 待付款`（允许重试支付）
   - 不更新 `paymentTime`

4. **支付超时（30分钟）**：
   - 支付单状态 = `3: 已关闭`
   - 订单状态 = `0` → `4: 已取消`
   - 恢复库存

### 4.2 退款相关

1. **申请退款时**：
   - 订单状态 = `1/2/3` → `5: 退款中`
   - `refundStatus` = `0` → `1: 申请中`
   - `refundApplyTime` = 当前时间
   - 记录原订单状态（用于拒绝时恢复）

2. **同意退款时**：
   - 订单状态 = `5: 退款中` → `6: 已退款`
   - `refundStatus` = `1` → `2: 已同意`
   - `refundProcessTime` = 当前时间
   - 恢复库存和销量
   - 执行实际退款（调用支付接口）

3. **拒绝退款时**：
   - 订单状态 = `5: 退款中` → 恢复原状态（`1/2/3`）
   - `refundStatus` = `1` → `3: 已拒绝`
   - `refundProcessTime` = 当前时间
   - `refundRejectReason` = 拒绝原因

### 4.3 超时处理

1. **待付款超时（30分钟）**：
   - 定时任务扫描 `status=0` 且 `createTime < 当前时间-30分钟` 的订单
   - 自动取消订单：`status=0` → `status=4`
   - 关闭支付单：`payment.status=0` → `payment.status=3`
   - 恢复库存

2. **待收货超时（7天）**：
   - 定时任务扫描 `status=2` 且 `shippingTime < 当前时间-7天` 的订单
   - 自动确认收货：`status=2` → `status=3`
   - 更新 `completionTime`

## 五、前端显示规则

### 5.1 订单状态显示

| 订单状态 | 显示文本 | 颜色 |
|---------|---------|------|
| `0: 待付款` | 待付款 | 橙色 |
| `1: 待发货` | 待发货 | 蓝色 |
| `2: 待收货` | 待收货 | 绿色 |
| `3: 已完成` | 已完成 | 灰色 |
| `4: 已取消` | 已取消 | 灰色 |
| `5: 退款中` | 退款中 | 黄色 |
| `6: 已退款` | 已退款 | 红色 |

### 5.2 支付方式显示

- **待付款订单**：
  - 如果 `paymentMethod` 有值：显示"支付宝"或"微信支付"
  - 如果 `paymentMethod` 为空：显示"待选择支付方式"（不应该出现，创建订单时必须选择）

- **已付款订单**：
  - 显示实际支付方式（从 `paymentMethod` 读取）

### 5.3 操作按钮显示规则

#### 用户视角

| 订单状态 | 显示按钮 |
|---------|---------|
| `0: 待付款` | 立即支付、取消订单 |
| `1: 待发货` | 申请退款 |
| `2: 待收货` | 确认收货、申请退款、查看物流 |
| `3: 已完成` | 评价、申请退款（限时7天）、查看物流 |
| `5: 退款中` | 查看退款进度 |
| `6: 已退款` | 查看退款详情 |

#### 商家/管理员视角

| 订单状态 | 显示按钮 |
|---------|---------|
| `0: 待付款` | 无操作（等待用户支付） |
| `1: 待发货` | 发货 |
| `2: 待收货` | 查看物流 |
| `3: 已完成` | 查看物流 |
| `5: 退款中` | 处理退款（同意/拒绝） |
| `6: 已退款` | 查看退款详情 |

## 六、数据库迁移方案

### 6.1 订单状态值调整

**问题**：当前 `5: 已退款` 需要改为 `6: 已退款`，新增 `5: 退款中`

**方案**：
1. 先新增状态值 `6`，将现有 `status=5` 的订单迁移到 `status=6`
2. 将 `refundStatus=1` 且 `status` 在 `1/2/3` 的订单，改为 `status=5`
3. 更新代码中的常量定义

### 6.2 支付方式字段处理

**问题**：创建订单时 `paymentMethod` 可能为空

**方案**：
1. 前端创建订单时必须选择支付方式
2. 后端验证：如果 `paymentMethod` 为空，返回错误
3. 历史数据：如果 `paymentMethod` 为空且 `status=0`，显示"待选择支付方式"

## 七、实施步骤

### 阶段一：状态定义调整（向后兼容）
1. 在 `Order.java` 中新增常量 `STATUS_REFUNDING = 5`，`STATUS_REFUNDED = 6`
2. 保留旧常量但标记为 `@Deprecated`
3. 更新状态判断逻辑，兼容新旧状态值

### 阶段二：退款流程改造
1. 申请退款时，将订单状态改为 `5: 退款中`
2. 处理退款时，根据结果更新为 `6: 已退款` 或恢复原状态

### 阶段三：支付方式处理
1. 前端创建订单时强制选择支付方式
2. 后端验证支付方式必填
3. 前端显示时，待付款订单显示选择的支付方式

### 阶段四：超时处理（定时任务）
1. 实现订单超时取消定时任务（30分钟）
2. 实现自动确认收货定时任务（7天）

### 阶段五：数据迁移
1. 迁移现有 `status=5` 的订单到 `status=6`
2. 迁移 `refundStatus=1` 的订单到 `status=5`

## 八、待讨论问题

1. **退款中状态是否必要？**
   - **支持**：用户能直观看到订单正在退款，体验更好
   - **反对**：增加状态复杂度，可以用 `refundStatus` 判断
   - **建议**：采用退款中状态，因为这是用户最关心的信息

2. **支付超时时间设置？**
   - 建议：30分钟（可配置）

3. **自动确认收货时间设置？**
   - 建议：7天（可配置）

4. **已完成订单退款时限？**
   - 建议：7天内可申请退款

5. **支付失败后是否允许重试？**
   - 建议：允许，但限制重试次数（如3次）
